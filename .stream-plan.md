## 1. Code-reality survey

### 1a. v29 wire surface I consume (already vendored in `codec/generated`)

Confirmed present in `src/core/daemon/codec/generated/daemon_api_client_types.h`:
- `tool_info { name, ?description, enabled: bool, ?requires }` and `response_tools` (Tools →
  `[* tool_info]`), plus `request_tool_list` ("ToolList", no session arg → node-wide inventory).
- `request_fingerprint_list { session }`, `request_fingerprint_revoke { session, fingerprint }`,
  `remembered_fingerprint { fingerprint, ?label }`, `response_fingerprints` (Fingerprints →
  `[* remembered_fingerprint]`).
- `ApprovalDecide_reason` (optional) on the decide request; `Approved_reason` (optional) on the
  inline `Respond` host-response body. **Both** deny paths can carry a reason at v29.
- `ApprovalInfo` (durable inbox row) carries: `session`, `request_id`, `prompt` (the node's honest
  free-text prompt), `?path`, `?fingerprint` (hex sha256, wire v28). **No** structured

**`ToolRegister` reality (decisive for D2):** the wire has `request-tool-register = { "ToolRegister":
{ "tool": tool-info } }`, but the node's `ControlApi::tool_register` is **only the default trait
impl returning `ApiError::Unsupported("tool_register")`** — no host override anywhere in
`daemon-node` (verified: `rg 'fn tool_register'` finds only the `lib.rs` default). Its semantics are
"register a (new) tool" (takes a full `ToolInfo`), **not** "toggle an existing tool's enabled
state". **Conclusion: D2 is read-only inventory + requirements guidance at v29.** Any
enable/disable/toggle op is a **deferred node-first follow-up** (§7). We do **not** fake
client-side toggles.

## 2. D2 — Tools settings panel (read-only inventory)

### Design

A node-wide (no session arg) inventory rendered from `ToolList → Tools`. Each row: tool name +
description, an **enabled/disabled** state, and — when disabled — the **unmet requirement**
(`requires`, e.g. `"tavily"`, `"vision"` feature, an MCP server name) with honest copy that does
**not** claim the app can fix it. No toggles (see §1a). Where the requirement is a credential the
app *can* set (e.g. a Tavily/Firecrawl key), the copy points at the existing Connection/credential
surface rather than duplicating key entry inline (keeps D2 scoped; inline key entry is a possible
enhancement flagged to the coordinator, §8-Q3).

### Shared model / facade (parity core)

New read-only facade `IToolInventory` (`src/core/tools/itool_inventory.h`) with a `VariantListModel`
`tools` + `int count` + `Q_INVOKABLE void refresh()`; rows:
`{ name, description, enabled(bool), requires(string, empty when none), requirementLabel(string
human copy) }`. `requirementLabel` is presentation formatting only (map a known `requires` token to
a friendly sentence; unknown tokens shown verbatim) — no domain logic.

- Daemon impl `DaemonToolInventory` (`src/core/tools/daemon_tool_inventory.{h,cpp}`) backed by a
  new `ToolRepository` (`repositories.{h,cpp}`, mirrors `ApprovalRepository`): `refreshTools()` →
  `encodeToolListRequest()`; `handleResponse` → `decodeTools(...) -> QList<DecodedToolInfo>`;
  `toolsRefreshed()`. **Not cached** (live inventory; matches approvals). `DecodedToolInfo {
  name, description, enabled, hasRequires, requires }`.
- Mock impl `MockToolInventory` (`src/core/tools/mock_tool_inventory.{h,cpp}`) for dev/tests
  (a handful of enabled + one requirement-gated row).
- Wire in `app_service_graph.cpp` (`graph.toolRepository` + `graph.tools`), expose as the `Tools`
  QML context property in `application.cpp`, refresh on connect-ready (alongside
  `approvals->refreshPending()`).

### GUI

New `Pages/ToolsSettingsSection.qml` + a `{ id:"tools", label: qsTr("Tools"), icon:
FontIcons.fa_wrench }` rail entry + `Component`/`Loader` case in `SettingsPage.qml`. Rows use Kit
controls: name + description, an enabled/disabled state pill (Kit `Chip`, tone `ok` when enabled /
`muted` when not), and a `requirementLabel` line + a "Set up…" affordance only when the requirement
is credential-shaped (deep-link to Connection). Empty/loading states honest.

### TUI

New Hub tab `Tools` (mirror `buildApprovalsMarkdown` → `buildToolsMarkdown` in `hub_*` + a
`TabModel::Tools` entry + parity route). Read-only list: `✓ name — description` /
`✗ name — needs <requirement>`. No mutation keys. Registered in `tui_page_hub.{h,cpp}` +
`tui_parity_routes.cpp` and fed the same `IToolInventory` via `root_widget_wiring.cpp`.

### Tests

`tests/unit/tst_tool_inventory.cpp` (decode `ToolInfo` enabled/requires; facade row projection;
requirementLabel mapping). Reuse a canned `Tools` CBOR fixture (hand-encode via the app encoder +
round-trip, mirroring `tst_approval_wire.cpp`). TUI markdown covered by parity harness.

## 3. D3 — structured approval prompts + deny-with-reason

Two surfaces share the model layer; both get the same upgrade. **Honesty constraint:** the wire
gives us the node's *honest prompt* (a multi-line string already containing resolved binary, argv,
cwd, exec-surface, digest) + the `fingerprint` hex + `path`. We render that faithfully; we do
**not** parse the prompt into fields or re-derive structure (that would fork the node's display).
"Structured" here = *layout* (monospace, line-preserving, not elided) + the fingerprint promoted to
a chip + scope options + reason input — using the fields the node actually sends.

### 3a. Facade / codec additions (I own these blocks)

- `DecodedApprovalInfo`: add `QString fingerprint;` (the full hex; keep `hasFingerprint`).
  `mappers.cpp`/`decode_responses.cpp` fill it from `ApprovalInfo.fingerprint`.
- `DecodedLogEntry`: no new wire field available for the inline park (no fingerprint on the stream);
  the honest prompt text is already in `hostPrompt`. Inline card renders `hostPrompt` structurally.
- `encodeApprovalDecideRequest(... , const QString& reason = QString())` — append optional reason;
  emit `ApprovalDecide.reason` only when non-empty (and only on a deny, matching the story). Tagged
  `// [wave2:app-approvals-safety]`.
- `encodeRespondApprovalRequest(... , const QString& reason = QString())` — append optional reason;
  emit `Approved.reason` when non-empty. Tagged.
- `ApprovalRepository::decide(session, requestId, allow, allowPermanent, reason=QString())` +
  `IApprovalsInbox::deny(id, reason=QString())` + `DaemonApprovalsInbox::deny` threads it through.
- `DaemonTurnEngine::respondApproval(requestId, allow, allowPermanent, reason=QString())` +
  `EditorController::answerToolApproval(..., reason=QString())` thread the reason on the inline
  deny path.

### 3b. Structured detail (both surfaces)

- Render the honest prompt as a **monospace, wrapped, line-preserving** block (not the current
  one-line elided cell). `path` shown as a labelled row when present.
- **Fingerprint chip**: when `fingerprint`/`hasFingerprint`, a Kit `Chip` (icon
  `fa_fingerprint`, short digest e.g. first 12 hex + "…", full value on tooltip/copy) with copy that
  states scope: allowing permanently remembers *exactly this command*.
- **Scope options**: Approve (allow once) + Allow permanently (only when the node offered it) +
  Deny — already present; keep, restyle into the structured layout.
- **Diff/args honesty**: `ApprovalInfo` carries **no diff or structured argv** field (the fs-edit
  prompt is only an 80-char preview node-side; the real diff exists only in the post-approval tool
  *result*). So a rich diff-in-approval is **wire-blocked** → recorded as a deferred node-first
  follow-up (§7). Where the tool *result* later carries a diff, the existing D1 `DiffBlock`
  renderer already handles it in the finished tool card — we do not duplicate it in the prompt.

### 3c. Deny-with-reason UI

- **GUI inbox** (`ApprovalsPage.qml`): a "Deny" that reveals an inline reason field (Kit field) +
  "Send deny"; empty reason = plain deny. Wired to `Approvals.deny(id, reason)`.
- **GUI inline** (`ToolApprovalBar.qml`): same pattern — Deny expands a small reason input; submit
  → `answerToolApproval(..., "denied", false, reason)`. Keeps the compact card by default.
- **TUI inbox** (`hub_ops`/`tui_page_hub`): `d` denies; a `D` (or `d` then reason prompt) opens a
  one-line reason entry (reuse the composer/free-text entry pattern) then denies with reason.
- **TUI inline** (`transcript_render`/`transcript_view`): the `Deny` control, when activated, drops
  into a `Freeform`-style reason line (reuse the existing clarify free-text control machinery) then
  emits `approvalDecided(callId, "denied", false, reason)`. `Control` gains an optional reason
  carry; `activateControl` (transcript_view.cpp) + `root_widget_input.cpp` dispatch updated
  (tagged).

### 3d. C5 insertion point for app-engines (origin chip) — STABLE, TAGGED

The engines sibling (C5) will insert a minimal origin-chip line ("requested by **gemini** via ACP")
into the approval card. I expose **one stable contract on each surface**, empty by default:

- **GUI inline** — in `ToolApprovalBar.qml`, a dedicated `Loader` at the top of the content Column,
  guarded by `visible: !!toolData.approvalOrigin`, sourcing an (initially empty) `originChipSlot`
  component, reading `toolData.approvalOrigin` / `toolData.approvalOriginKind`. Tagged
  `// [wave2:app-approvals-safety] C5 origin-chip insertion point (app-engines fills the field +
  component)`. The engines sibling only needs to (a) populate `approvalOrigin*` upstream in their
  stream mapping and (b) fill `originChipSlot` (a Kit `Chip`). No structural churn to my code.
- **GUI inbox** — in `ApprovalsPage.qml`, the same slot above the prompt block, keyed on
  `entry.origin`. `DecodedApprovalInfo`/row projection leaves an `origin` field defaulting empty
  for engines to populate (the wire may later add attribution; today it is empty and the chip
  hidden).
- **TUI** — in `emitApprovalBar`, a tagged no-op guard `if (!origin.isEmpty()) emit an origin line`
  reading a `view["approvalOrigin"]` metadata key (empty today).

I will document this contract in the plan and in code comments so the engines sibling's insertion
is a field-populate + component-fill, never a rewrite of the card.

## 4. D4 — remembered-fingerprint management (list + revoke)

### Placement (proposal — coordinator decision flagged, §8-Q1)

Fingerprints are **strictly per-session** (both wire ops require a `session`). The global
`Settings → Safety` section has **no active session** to scope to, so the honest primary home is the
**per-session surface**:

- **Primary: session settings popover** (`SessionSettingsPopover.qml`) — a new "Remembered
  approvals" subsection listing the active session's fingerprints with a per-row Revoke, matching
  TOOL-5's "listable/revocable in the session settings popover". The `ISessionSettings` facade
  already carries the active `sessionId`, so the list is naturally scoped.
- **Secondary: `SafetySettingsSection.qml`** gets a short, honest pointer paragraph ("Remembered
  command approvals are managed per session from the composer's session settings") — mirroring how
  it already points at the per-session approval mode. No global list there (nothing to scope).

### Shared model / facade

New read-only+revoke facade for the active session. Two viable shapes (coordinator pick, §8-Q2):
(a) extend `ISessionSettings` with a `fingerprints` `VariantListModel` + `revokeFingerprint(fp)` +
`refreshFingerprints()`, reusing its `sessionId`; or (b) a standalone `IRememberedFingerprints`
facade + `FingerprintRepository`, driven by the composer's active session. **Recommendation: (a)** —
fewer new context properties, and the popover already binds `SessionSettings`; the repository work
is identical either way.

- New `FingerprintRepository` (`repositories.{h,cpp}`, mirrors `ApprovalRepository`):
  `refreshFingerprints(session)` → `encodeFingerprintListRequest(session)`; `handleResponse` →
  `decodeFingerprints(...) -> QList<DecodedRememberedFingerprint>`; `revoke(session, fingerprint)`
  → `encodeFingerprintRevokeRequest(...)`, on Ok re-list. Not cached. `DecodedRememberedFingerprint
  { fingerprint, hasLabel, label }`.
- **Provenance honesty:** the DTO carries only `fingerprint` (hex) + optional `label` (always
  `None` today node-side). **No timestamp, no command text.** So the UI honestly shows: the short
  digest (full on copy), the `label` when present (future), and — nice-to-have — a "currently
  pending in the inbox" correlation when the same hex also appears in `Approvals.pending` (both
  come from the same session). It must **not** fabricate "allowed at <time>" or a command summary.
  A richer provenance (label/command-summary/timestamp on `RememberedFingerprint`) is a deferred
  node-first follow-up (§7).

### GUI

Session settings popover subsection: `SectionLabel`-style header + a `Repeater` of fingerprint rows
(Kit `Chip` digest + Revoke `TextButton` with a confirm), empty state "No remembered approvals for
this session." Refresh on popover open and after a revoke.

### TUI

The session-settings overlay (`settings_editor` / composer session settings surface) gains a
"Remembered approvals" block listing digests with a Revoke key, scoped to the active session
(same facade). Parity-covered.

### Tests

`tests/unit/tst_fingerprint_manage.cpp` (encode list/revoke; decode `RememberedFingerprint`;
facade list + revoke-then-refresh; label-absent honesty). Popover/overlay render covered in
`tests/qml`/`tests/tui`.

## 7. Deferred node-first follow-ups (recorded, not implemented)

1. **Tool enable/disable mutation op** — `ToolRegister` is `Unsupported` and is a *register* verb,
   not a toggle. A real per-tool enable/disable (session- or node-scoped) needs a new node op; D2
   stays read-only until then.
2. **Diff-in-approval** — `ApprovalInfo` carries no diff/structured-argv; a rich diff review inside
   the approval prompt needs the node to attach the pending diff to the park payload.
3. **Fingerprint provenance enrichment** — `RememberedFingerprint` has only `fingerprint` + always-
   `None` `label`; "what command / when allowed" needs a node-side label/command-summary/timestamp
   (the `label` field is already reserved on the wire, so a future fill needs no bump).

## Cache-schema impact (`client_cache_schema.h`, current `kSchemaVersion = 7`)

- **D1 reason/message/fatal**: `connection`/`presence` are cached on `daemon_transport_instances`
  for offline-first rendering. For parity (and to gate the re-auth affordance offline-first),
  reason/message/fatal should be cached too → **schema v8**: add `connection_reason TEXT`,
  `connection_message TEXT`, `fatal INTEGER DEFAULT 0` columns + `CachedTransportInstanceRow` fields
  + `daemon_cache_store.cpp` upsert/read. Non-authoritative → drop+rebuild. (Open Q1 for in-memory
  alternative.)
- **Stretch (endReason on fleet units)**: adds `end_reason TEXT` to `daemon_fleet_units` +
  `CachedFleetUnitRow.endReason` + decode plumb (`DecodedUnitNode.endReason` already exists) — also a
  v8 column. If folded in, it shares the single v8 bump (kept a separate, strike-able plan item).
- Everything else (tools, approvals detail, fingerprint provenance, provider sign-in, foreign
  failure) is **live/in-memory** (ToolRepository/ApprovalRepository/FingerprintRepository/turn engine
  are not cached) → **no schema change**.

## Open questions for the coordinator (each with a recommendation)

1. **Cache reason/message/fatal (v8) vs in-memory-only?** Recommendation: **cache them (v8 bump)** —
   consistent with connection/presence being offline-first and lets `fatal` gate re-auth offline. If
   coordinator prefers minimal churn, keep them in-memory on the transport-registry row (no schema
   bump) — reason/message/fatal then blank until the first live TransportChanged/instances refetch.
2. **Remove semantics**: today's Channels "Remove" drops the *stored credential* (`Accounts.remove` /
   CredentialRemove). v30 `TransportRemove{transport}` removes the *transport instance*. Recommendation:
   the Channels Remove action sends **`TransportRemove`** (transport-scoped, destructive, confirmed);
   keep credential removal as a distinct lever on `AccountsPage` (different object). Confirm we should
   not also chain a credential remove.
3. **TUI Tools read-write interaction**: the TUI tools inventory renders inside the Settings-page
   markdown with no per-row selection. Recommendation: add tool-row selection + a toggle key within
   the Settings tools block (parity with Approvals selection). Alternative: a dedicated Tools overlay.
   Confirm preferred model.
4. **Approval diff parse reuse**: `parseDiff`/`DiffBlock` live in the block editor; ApprovalsPage has
   no `EditorController`. Recommendation: factor a controller-free diff-line parse (shared util or a
   `Q_INVOKABLE` on the approvals inbox) so `DiffBlock` renders in Settings/Approvals without pulling
   the whole block-editor controller. Confirm acceptable vs. instantiating an EditorController there.
5. **ConvList polling retirement scope**: OK to remove the TUI tab-enter refetch-all and the GUI
   per-expand refetch, relying on `ConversationsChanged`/`MembershipChanged` + the connect-ready
   baseline + reconnect seed? (Recommendation: yes.)
6. **Stretch (endReason on `CachedFleetUnitRow`)**: fold into the same v8 bump, or leave out?
   Recommendation: fold in only if the Wave A "error strip" actually needs the offline-first value;
   otherwise strike (keep this a separate, isolatable plan item).

## 0. Code-reality survey (what exists today, verified)

### 0.1 The v29 wire surface is already in the generated codec (do NOT hand-edit)
`src/core/daemon/codec/generated/daemon_api_client_types.h` already carries every v29 type this
stream needs:
- `completion_notice_ref { child (session-id tstr), ? call_id (tstr/null) }` and
  `user_msg { text, ? attachments, ? notice }` where `notice = completion_notice_ref / null`
  (types.h ~L67–98). **Note (from the node landing record): `notice` has no
  `skip_serializing_if`, so the node ALWAYS emits `"notice": null` on every StartTurn/user-msg** —
  the decoder must treat `user_msg_notice_present == true` with
  `choice == user_msg_notice_null_m_c` as "no notice", and only a
  `user_msg_notice_completion_notice_ref_m_c` as a real delegation notice.
- `unit_node` gained `unit_node_lifetime` (`delegation_lifetime = "Persistent"/"Ephemeral"`, null-able)
  and `unit_node_engine` (`engine_selector`, null-able) after the existing `unit_node_role`
  (types.h ~L3874–3900).
- `engine_selector = "Core" / { "Foreign": { "agent": tstr } }` — the v23 `Acp` variant was renamed
  to `Foreign` (types.h ~L765–784). The Rust `caps_report` encoder is present
  (`daemon_api_client_encode.c` ~L10666) so a `Caps`/`CapsReport` decode path can be wired.

### 0.2 The fleet mapper decodes UnitNode but NOT the new fields
`codec_detail::decodeUnitNodeStruct(const unit_node&)` (`node_api_codec/mappers.cpp:133–163`) stops
at `role`; it does **not** read `unit_node_lifetime` or `unit_node_engine`. The facade struct
`DecodedUnitNode` (`node_api_codec.h:608–621`) has `id/kind/state/endReason/work/children/
profileRef/sessionId/title/role/parentId/depth` — no `lifetime`, no `engine`. The tree decode/flatten
path is `fs_fleet_channels.cpp:decodeTreeReport` → `flattenTreeNodes` → `decodeUnit`.

### 0.3 The wave-1 F3 engine chip is a client-side profile-join (to be replaced)
`DaemonFleetTree` (`src/core/daemon/daemon_fleet_tree.{h,cpp}`) projects `FleetRepository::cachedUnits()`
into the `IFleetTree` rows. The engine chip (C3/ENG-7) is computed by **joining** each row's
`profileRef` against `IProfileStore::profile(ref)` to read `engine` (`"Core"|"Acp"`) + `acpAgent`
(`daemon_fleet_tree.cpp:71–81`); the join re-runs on `IProfileStore::changed` (async ProfileGet
hydration). Degrades to `"Core"` when the store is absent or the profile spec has not hydrated.
The profile decode itself (`mappers.cpp:538–546`) reads `engine_selector` and maps the (renamed)
`engine_foreign_m` arm to the app-internal label **`"Acp"`** — i.e. the app kept "Acp" as its
engine-kind discriminator string through the wire rename; the FleetPage/TUI chips key on
`engine === "Acp"`.

### 0.4 The offline-first fleet cache has no lifetime/engine columns
`daemon_fleet_units` (`client_cache_schema.h:79–94`, schema **v6**) stores
`unit_id/parent_id/depth/ordinal/name/kind/state/role/profile_ref/session_id/work` — no lifetime,
no engine. Today the engine chip survives offline only because the *profile* cache
(`daemon_profiles`) + the join reconstruct it. The cache is explicitly non-authoritative (bumps
just drop+rebuild).

### 0.5 The FleetChanged/tree-refetch seam
`SubscriptionManager::applyEvent` (`subscription_manager.cpp:111`) routes `NodeEvent`s.
`DecodedNodeEvent::Kind` (`node_api_codec.h:573–588`) has **no `FleetChanged` arm today**; the fleet
tree is refetched only via `ResyncNeeded` → `m_fleet->refreshTree("subscription")` (L163). The
`node_event_fleet_changed` encoder exists in the generated codec but nothing decodes it into a
`DecodedNodeEvent` yet.

### 0.6 How the injected completion notice renders today (F1/F2)
The detached completion notice is injected by the node as a fresh user turn. The app sees it on the
merged session log as an **Inbound Command echo**: `decode_responses.cpp:280–282` decodes
`StartTurn.input.user_msg_text` into `DecodedLogEntry.commandText` (and Steer at L283–284). It does
**not** decode `user_msg.notice`. `DaemonTurnEngine::applyLogPage` (`daemon_turn_engine.cpp:475–495`)
then emits a plain `{type:"userMessage", text}` event and caches a `Message`/`User` transcript block.
So `[subagent {child} completed] {summary}` renders as ordinary user text — no linkage to the
spawning `orchestrate` call.

### 0.7 How the orchestrate spawn renders today (F1)
`orchestrate` is rendered as a generic ToolCall block. Tool blocks flow ToolStarted/ToolFinished →
`toIngestEvent` → BlockEditor. The view fields are built by `be::buildToolView`
(`BlockEditor/core/agent_block.cpp:319–392`): `variantForName(name)` returns
`"clarify"|"image-generate"|"generic"` (L20–29), and `projectToolDetail(detailKind, detailBody,
summary)` maps rich detail into renderer keys (D1). `ToolCallBlock.qml` renders a card whose body
routes by `variant` (`clarify` → interactive) and `detailKind` (`diff`/`search-results`/`image`/
`ansi-stream`/… → sub-renderer). D1 already plumbs `detailKind` + `detailBody` (JSON) end-to-end
(live + journal + cold-start) and persists them on the transcript cache row (`detail_kind`,
`detail_body`; `daemon_cache_store.h:97–114`, schema v6).

### 0.8 The live subagent strip (context, not the transcript)
`SubagentModel` (`Session/subagent_model.{h,cpp}`) holds live rows keyed by id

**not** the transcript delegation card.

### 0.9 Operator steer/cancel + drill-in (existing seams)
FleetPage rows already offer operator **Steer…**/**Cancel** on delegated children (F4, wave-1):
`SessionRoster.steer/startTurn/interrupt(sessionId)` (`FleetPage.qml:198–219`), TUI `t`/`c` keybinds

### 0.10 There is no Caps facade / repository (F7)
No `DecodedCapsReport` in `node_api_codec.h`, no `CapsRepository`, no `Caps` encoder call in

## 1. Per-item design

### F3 — tree chips from the wire (lifetime + engine)  ← do first (unblocks the seam)

**Decision — WIRE, drop the client-side engine join.** The app peer is v29 and
`WireVersion::is_compatible` is strict equality, so there is no older peer to fall back for. The
wire `unit_node.engine` is strictly better than the wave-1 profile-join: it is **per-child**
(a child under a foreign-engine profile — ENG-2 — reports its own engine even when the profile is
unknown / falls back), and `unit_node.lifetime` has no join equivalent at all. So:

1. **Facade decode (mine).** Extend `DecodedUnitNode` with `QString lifetime` (""/"Persistent"/
   "Ephemeral"), `QString engineKind` (""/"Core"/"Acp"), `QString engineAgent`. In
   `decodeUnitNodeStruct` (`mappers.cpp`), after the `role` block, decode
   `unit_node_lifetime`/`unit_node_engine` when present+non-null. Reuse a small shared helper
   `decodeEngineSelector(const engine_selector_r&, QString* kind, QString* agent)` — factor it out
   of the existing profile-spec decode at `mappers.cpp:538–546` so profile-engine and unit-engine
   decode identically (map the `engine_foreign_m` arm → kind `"Acp"` + agent, keeping the app's
   existing `"Acp"` label for zero cross-file churn — see the naming note below). Tag the factored
   helper `// [wave2:app-delegation]`.
2. **Cache columns (schema bump — see §4).** Add `lifetime`, `engine`, `engine_agent` to
   `daemon_fleet_units` so the chips survive offline-first (parity with today's offline engine
   chip). Populate from the enriched `DecodedUnitNode` in `FleetRepository::syncFleetUnits`
   (`repositories.cpp`) and read back into `CachedFleetUnitRow`.
3. **View-model (`DaemonFleetTree::rebuild`).** Set `row["lifetime"]` from the unit; set
   `row["engine"]`/`row["acpAgent"]` from the unit's wire fields when present, else fall back to the
   existing profile-join (kept as a cheap degrade for units the node did not enrich, e.g. the
   in-memory `project_unit` builder). If the wire always populates engine for bound children, the
   join becomes dead for engine and the `IProfileStore` coupling can be dropped in a follow-up; I'll
   keep it as the fallback this wave to avoid a regression on un-enriched rows.
4. **GUI (`FleetPage.qml`).** Add a lifetime `Kit.Chip` next to the engine chip: `Persistent`
   (muted, `fa_thumbtack`/`fa_link`) vs `Ephemeral` (accent/`fa_hourglass`), `visible:
   entry.lifetime !== ""`. Engine chip binding unchanged (still keys on `entry.engine === "Acp"`).
5. **TUI (`hub_ops.cpp buildFleetMarkdown`).** Add a `lifetimeToken(n)` helper mirroring the
   existing `engineToken(n)`, appended to the row line (e.g. `· persistent`). Parity with the GUI
   chip.

**Naming note (coordination with app-engines):** the app currently uses `"Acp"` as its engine-kind
label everywhere (profile decode, FleetPage/TUI chip bindings, `DecodedProfileSpec.engineKind`).
The wire renamed the variant to `Foreign`. I will keep the app-internal `"Acp"` label for F3 to
avoid a repo-wide rename that overlaps the app-engines stream (which owns the Agent* registry). If
the coordinator wants the UI relabeled `Foreign`, that is a separate app-engines change; I decode
the `engine_foreign_m` arm either way.

  "delegation"`) rendering child identity chips (engine/lifetime — resolved by a lookup into the

**(b) Completion-notice linkage (F2 consumption).**

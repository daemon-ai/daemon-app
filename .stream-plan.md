## 1. Facade-rename proposal — **RECOMMEND RENAME NOW** (`Acp*` → `Agent*`, `engineKind "Acp"` → `"Foreign"`)

**Recommendation: rename the app facade to match the wire, and change the `engineKind` sentinel
`"Acp"`→`"Foreign"`, carrying ACP-vs-stream-json as a separate `protocol` field.** Rationale:

1. **Correctness, not cosmetics.** v29 made `Acp` a *protocol* (`agent_protocol = Acp|StreamJson`), not
   the engine kind — the wire engine selector is `Foreign{ agent }`. Keeping `engineKind == "Acp"` now
   actively lies for a stream-json foreign engine (ENG-5). The chip label "…(ACP)" must come from the
   catalog entry's `protocol`, not from the engine kind.
2. **I own this vertical.** ENG-5 is the largest rename surface and it is entirely engine-identity code
   (C3 chips + the picker + the repo) — my domain. Doing it piecemeal later means a second pass over the
   same files.
3. **Fallout is bounded and mechanical** (full list below). No behavior change beyond the label source.

**Fallout list (all become `// [wave2:app-engines]`-tagged edits):**
- `node_api_codec.h`: `DecodedAcpAgentEntry`→`DecodedAgentEntry` (+ new `protocol`, `capabilities?` fields);
  `engineKind` default/comment `"Acp"`→`"Foreign"`; method decls `encodeAcpCatalogRequest`→
  `encodeAgentCatalogRequest`, `encodeAcpDiscoverRequest`→`encodeAgentDiscoverRequest`, `decodeAcpCatalog`→
  `decodeAgentCatalog`; `ApiResponseKind::AcpCatalog`→`AgentCatalog`.
- `node_api_codec/mappers.cpp:541,1030` : `engineKind = "Foreign"`; foreign-agent get/set arms.
- `node_api_codec/decode_responses.cpp:75,1091` : rename + decode `protocol`.
- `node_api_codec/encode_requests.cpp:312,989` : rename.
- `repositories.{h,cpp}`: `AcpRepository`→`AgentRepository`, correlations `repo/acp-*`→`repo/agent-*`.
- `app_service_graph.{h,cpp}`: `graph.acp`→`graph.agents`; QML context property `AcpAgents`→`Agents`.
- `iprofile_store.h` + `daemon_profile_store.{h,cpp}`: `createAcpProfile`→`createForeignProfile`;
  `engineKind = "Foreign"`; row `engine` value `"Foreign"`.
- QML engine-chip sites: `ComposerControls.qml`, `ProfilesPage.qml`, `FleetPage.qml` — `=== "Acp"` →
  `=== "Foreign"`; label suffix from `protocol`. `AgentTypePicker.qml` (`AcpAgents`→`Agents`,
  `acpRows`→`agentRows`), `NewAgentDialog.qml` (`createAcpProfile`→`createForeignProfile`).
- TUI: `composer_chrome.cpp:159`, `hub_profiles.cpp:13` (`== "Acp"`→`"Foreign"`, protocol-aware label),
  `agent_type_view.{h,cpp}` (`AcpRepository`→`AgentRepository`), `tui_dialogs.{h,cpp}`,
  `root_widget_input.cpp` (`m_services.acp`).
- Tests: `tst_daemon_fleet_tree.cpp`, `tst_first_run_model.cpp` reference engine strings; update.

**Migration nuance (call out to coordinator):** the profile's wire engine selector carries only
`Foreign{ agent }` — **not** the protocol. To render "…(ACP)" vs "…(stream-json)" on a profile/session/
fleet chip requires a second join (agent name → catalog `AgentRepository` entry → `protocol`). The chip's
primary signal stays the **agent name**; the protocol suffix is best-effort when the catalog is loaded and
degrades to a bare "Foreign"/agent name otherwise. I'll expose a facade helper `protocolForAgent(name)`.

**If the coordinator vetoes the rename:** fallback is keep `AcpRepository`/`engineKind "Acp"` names but
STILL add the `protocol` field + register/remove methods under the old names, and gate the "(ACP)" label
on `protocol == "Acp"`. This leaves the codebase saying "Acp" where the wire says "Foreign" — I advise
against it, but it is a clean, smaller diff.

## 5. C6 — foreign failure states (within what the node reports today)

Survey → design, staged earliest-first (ENG-8: create > session-open > mid-turn > probe):

1. **Registration probe (ENG-3/ENG-8).** `AgentRegister` re-probes; the node's reply is `Ok` (entry
   persisted with probed `installed`/`version`) or `Error`. The register dialog's probe panel renders:
   - success → "handshake ✓ · ACP `<version>`" (from the refreshed catalog entry) or, for stream-json,
     "found on PATH" (PATH-only probe, no version — per the v29 plan §2f).
   - failure → surface the node's `Error` message via `operationFailed`, distinguishing **"binary not
     found"** from **"handshake failed/timeout"** *iff the node's error string distinguishes them*
     (see Risks — may be a deferred node-first ask if the string is opaque).
2. **Create-time (ENG-8, proven node-side).** Make `createForeignProfile` failures **field-level** instead
   of a post-close toast: `NewAgentDialog`/`AgentTypePicker` keep open and show the agent-named error.
   Requires routing `profileOpFailed` back into the dialog's accept gate (don't `openNewAgentChat()` until
   the create is acknowledged). This is a small state change in `NewAgentDialog.qml` +
   `DaemonProfileStore` (surface a per-op result/last-error). TUI mirrors in `tui_dialogs`.
3. **Session-open (agent uninstalled after creation).** `SessionCreate` → `Error("agent <name> is not
   installed")` already flows to the session/turn error path. Add a **session-level error banner** in the
   transcript/session surface with an **"Open Agents settings"** action (deep-link `Nav → settings#agents`,
   the C1 surface) so the user can re-scan/fix. Reuse the existing turn error state; add foreign grammar.
4. **Mid-turn death.** `TurnFinished{ end_reason: Failed }` already ends the turn with a failure summary
   (no silent hang) + keeps streamed content; retry re-spawns. Improve the copy for foreign sessions to
   name the agent + stage where the node message allows; keep the transcript.

**Banner grammar (vision `resolve | spawn | handshake | mid-turn`).** The node's error strings are
free-text today; the client can name the **agent** (from the profile) and echo the node message honestly,
but a *structured* stage code is not on the wire. **Deferred node-first follow-up (record, don't fake):**
a structured foreign-failure enum (`{stage, agent}`) on the error/TurnFinished for reliable stage badging,
plus a conformance test for mid-turn death (ENG-8 Reveals: none exists). Until then C6 renders the honest
node message + agent name + the jump-to-catalog action.

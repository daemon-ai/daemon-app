## 1. Kit conventions & the Chip foundation (item 1)

### Conventions established from code
- Kit lives in `src/DaemonApp/Controls/`, module `DaemonApp.Controls` (`qt_add_qml_module(da_controls STATIC URI DaemonApp.Controls …)` in `Controls/CMakeLists.txt`). Each control is a `.qml` file listed in `QML_FILES`. **No generated `qmldir` to hand-edit** — `qt_add_qml_module` produces it; adding a file to `QML_FILES` is the whole registration step.
- Controls import `DaemonApp.Theme` and bind **only** `Theme.*` / `FontIcons.*` tokens (`Controls/README.md`, `AGENTS.md` "UI kit & theming").
- The `QQC as QQC` wrapping pattern for controls that skin a `QtQuick.Controls` type (`TextButton.qml` wraps `QQC.Button`). A pure presentational chip needs no `QQC` base — model it as a themed `Rectangle` (like the existing feature-local `Composer/AttachmentChip.qml`, which is the de-facto chip today but lives outside the Kit).
- Icon primitive: `Controls/Glyph.qml` (a themed FontAwesome `Text`; `glyph`, `family` aliases). Reuse it inside the chip.
- **No controls "gallery"** app/page exists; controls are covered by QML tests under `tests/qml/*` (e.g. `tests/qml/theme/tst_theme.qml`, `tests/qml/composer/tst_composer.qml`) driven by `*_main.cpp` harnesses, plus `tests/unit/tst_theme_palette.cpp` for token coverage. There is no dedicated chip test yet.

### New file — `src/DaemonApp/Controls/Chip.qml`
Compact label + optional icon + optional action, kit-themed. Proposed surface:

```qml
// Chip.qml (DaemonApp.Controls)
Rectangle {
    property string text: ""
    property string iconGlyph: ""                 // FontAwesome glyph, "" = no icon
    property string iconFamily: FontIcons.faSolid // override for mtSymbols etc.
    property string tone: "neutral"               // neutral | accent | muted | danger
    property bool interactive: false              // whole chip is a button
    property bool closable: false                 // trailing xmark affordance
    property string tooltipText: ""               // Kit Tooltip when non-empty
    signal clicked()
    signal closeRequested()
    // radius: Theme.radius; border from Theme.border; bg from Theme.codeBackground
    // (tone → text/border/icon color mapping; hover/pressed via Theme.hover/pressed)
}
```

- Register: add `Chip.qml` to `Controls/CMakeLists.txt` `QML_FILES` (additive; only overlap with siblings is this one line — trivial merge).
- Theming: reuse `Theme.radius`, `Theme.border`, `Theme.codeBackground`, `Theme.text`, `Theme.textMuted`, `Theme.accent`, `Theme.hover`, `Theme.pressed`, `Theme.iconMuted`. **Open:** if design wants a tinted accent/danger *background* (not just border/text), that is a new `ThemePalette` token (`chipAccentBg`/`chipDangerBg`) added in `src/core/theme` + `Theme.qml` + `tui_palette`, covering all four themes (Light/Dark/Sepia/Midnight). Default plan: tint via border/text only, **no new token** (keeps the change minimal and 4-theme-safe). Decide in review.
- Optional (surgical, defer): refactor `Composer/AttachmentChip.qml` to consume `Kit.Chip` (`closable`). Not required for this wave; note it, don't force it.

### TUI parity for chips
The TUI renders pages as **markdown strings** (`src/tui/pages/hub_ops.cpp` builders, `ComposerChrome`), not QML controls. A "chip" in the TUI is therefore an **inline text token** in the relevant markdown builder (e.g. `⬡ gemini`, `Ask`, `⇄ #ops`). There is **no TUI Chip widget** to build; parity = emitting the token in each corresponding builder (called out per item).

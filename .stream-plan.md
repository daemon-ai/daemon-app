## 1. Code-reality survey

### 1.1 The event feed & dispatch (the spine of B5 + F5)
- One node-wide `EventsSince` push stream is owned by
  `SubscriptionManager` (`src/core/daemon/subscription_manager.{h,cpp}`). `onStreamItem`
  decodes an `EventsPage` and calls `applyEvent()` per `DecodedNodeEvent`.
- `applyEvent()` routes by `DecodedNodeEvent::Kind`:
  `RosterChanged`/`SessionMetaChanged` ‚Üí debounced `SessionRepository::refreshSessions()`
  **+ `FleetRepository::refreshTree("subscription")`** (a fleet unit IS a durable session);
  `ApprovalPending` ‚Üí inbox badge; `SessionAdvanced` ‚Üí focused-engine nudge;
  `DownloadProgress` ‚Üí `ModelRepository::applyDownloadProgress(...)` (**the in-place
  event-payload-apply precedent**); `CatalogChanged` ‚Üí catalog refetch;
  `ResyncNeeded` ‚Üí broad re-baseline; `Unknown` ‚Üí **no-op**.
- The codec facade `DecodedNodeEvent::Kind` enum (`node_api_codec.h`) has **no
  `FleetChanged` and no `TransportChanged`** today. `decodeEventsPage`
  (`node_api_codec/decode_responses.cpp` ~L312) switches on the generated
  `node_event_choice`; the `node_event_fleet_changed_m_c` and
  `node_event_transport_changed_m_c` arms fall through `default ‚Üí Kind::Unknown`
  (silently ignored). **So both are currently decoded-away.**
- The regenerated **v29 vendored codec already carries** `node_event_fleet_changed`,
  `node_event_transport_changed`, `connection_state_r`, `presence_state_r`
  (verified in `codec/generated/daemon_api_client_types.h`). My work is pure C++
  facade exposure ‚Äî no `just update-codec`, no generated/vendored edits.

### 1.2 Wire shapes (daemon-node `master` CDDL, v29)
- `node-event-fleet-changed = { "FleetChanged": { "rev": uint64 } }` ‚Äî **existed pre-v29**.
- `node-event-transport-changed = { "TransportChanged": { "transport": transport-id,
  "connection": connection-state, ? "presence": presence-state } }` ‚Äî **v29 (this stream)**.
- `connection-state = "Offline" / "Connecting" / "Connected" / "Error"`.
- `presence-state = "Unknown" / "Offline" / "Available" / "Idle" / "Away" / "Busy"`.
- **No `reason` field on `TransportChanged`.** The scope's "disconnected with reason if
  carried" is not satisfiable at v29 ‚Äî the wire carries only the coarse state. The chip
  shows the state; a reason string is a node-first follow-up (open Q4).
- Node emit source (per v29 landing record Q10): MVP emits `TransportChanged` only at
  **coarse real transitions** (serve start‚ÜíConnected, credential-removal/teardown‚ÜíOffline,
  error‚ÜíError, presence=Unknown). Full presence tracking is a node follow-on ‚Äî so the app
  must render the coarse truth and not imply finer granularity than the node reports.

- TUI `src/tui/pages/hub_channels.cpp` (`buildChannelsMarkdown`): same seams, rendered as
  markdown; every account shown expanded; status dot via `presence->connectionState(tx)`.
  Already read-only; connect deferred.
- `src/tui/root_widget_wiring.cpp` (~L163-192): the TUI Channels page **already
  re-renders on `ITransportRegistry::{adapters,instances,conversations}Changed` and on

### 1.4 Presence projection (B5 seam)
- `DaemonPresenceService` (`src/core/daemon/daemon_presence_service.{h,cpp}`) projects
  `TransportRepository::cachedInstances()` ‚Üí per-transport connection/presence and emits
  `presenceChanged(tx)`. It **rebuilds only on `TransportRepository::instancesRefreshed`**
  ‚Äî i.e. today presence is refresh/poll-derived, never event-driven. Its own doc-comment
  says "Real live presence is a deferred backend follow-up." **B5 delivers that.**

### 1.5 Transport repository & cache
- `TransportRepository` (`repositories.{h,cpp}`) issues `TransportInstances`/`ConvList`,
  writes `CachedTransportInstanceRow`/`CachedConversationRow` into `DaemonCacheStore`,
  emits `instancesRefreshed`/`conversationsRefreshed(tx)`. `syncTransportInstances` upserts
  live rows + prunes missing ones. `CachedTransportInstanceRow` already has
  `connection`/`presence` columns ‚Üí **no schema change for B5**.
- `client_cache_schema.h` is at **v6**. B5 patches existing columns; B2's "new-room"
  affordance is in-memory (¬ß4.2). **No `client_cache_schema.h` bump.**
- Codec `connectionStateName`/`presenceStateName` (lowercase mappers, e.g. `"connected"`,
  `"available"`) live file-local in `node_api_codec/fs_fleet_channels.cpp` and already
  match the GUI `dotColor()` / TUI `dotFor()` expectations.

### 1.6 Fleet refresh (F5)
- `FleetRepository::refreshTree(source)` issues `Tree`, decodes to `daemon_fleet_units`,
  emits `treeRefreshed()`. Today the tree only refreshes on roster deltas / `ResyncNeeded`
  (it rides `RosterChanged`), never on the dedicated `FleetChanged` (decoded-away). So a
  fleet mutation that does **not** move the roster (e.g. a subagent state change with no
  new session row) can currently go unrefreshed until the next roster event. F5 closes it.

  - phases: `idle/beginning/awaiting_browser/completing/success/failed/cancelled`.

- GUI sheet `src/DaemonApp/Pages/AuthFlowSheet.qml`: `openFlow(family, params, bindProfile)`
  (start a known family directly) and `openPicker(bindProfile)` (family dropdown +
  schema-driven param fields; e.g. Matrix SSO's `homeserver`). `AccountsPage.qml` already

### B1 ‚Äî Connect a Matrix account from Channels (GUI + TUI)

**Approach:** reuse the wave-1 auth stack end to end; add only the Channels entry point
and a single shared post-success refresh.

- **Shared (both surfaces), `app_service_graph.cpp`** ‚Äî the "land connected, roster
  populating" behavior lives once in the service graph, not per-surface:
  `connect(authFlowController, &AuthFlowController::succeeded, ‚Ä¶)` ‚Üí
  `transportRepository->refreshInstances()` and, when `transportInstance` is non-empty,
  `transportRepository->refreshConversations(transportInstance)`. Both surfaces then update
  through their existing `instancesChanged`/`conversationsChanged` reactivity; the incoming
  `TransportChanged` (B5) lights the status chip. Tagged, appended after
  `authFlowController` is constructed (~L339).
- **GUI, `ChannelsPage.qml`:** add an `AuthFlowSheet { id: authSheet }` instance. Change the
  per-adapter "Connect" button from `enabled:false` to
  `enabled: modelData.capabilities.interactiveAuth === true` and
  `onClicked: authSheet.openFlowForFamily(modelData.family)` (see minimal extension below).
  If `interactiveAuth` is false the button stays disabled with a tooltip explaining that
  family has no browser sign-in.
- **TUI, `root_widget_input.cpp` + `hub_channels.cpp`:** add `'c'` on `TabModel::Channels`
  ‚Üí `openAuthFlow()` (the existing shared launcher). Add a one-line hint to the Channels
  markdown footer (`"Press c to connect an account."`). The launcher's family-pick palette
  already narrows to matrix when it's the only interactive family.

**What B1 needs from AuthFlowSheet (minimal, tagged extension):**
The sheet already exposes everything functionally. To make the Channels "Connect" button
land directly on a *known* family (matrix) with its param form ‚Äî rather than the full
family dropdown ‚Äî add one small QML function:

```qml
// [wave2:app-channels-liveness] open the picker pre-narrowed to one family
function openFlowForFamily(family, bindProfile) {
    root.bindProfile = bindProfile || "";
    if (AuthFlow) { AuthFlow.reset(); AuthFlow.refreshProviders(); }
    root.refreshProviderRows();
    // preselect the row whose family matches; the existing idle form renders its params
    for (var i = 0; i < root.providerRows.length; ++i)
        if (root.providerRows[i].family === family) { root.pickedProvider = i; break; }
    root.forcedFamily = family;   // hides the provider Dropdown when set (visible:!forcedFamily)
    callbackField.text = ""; open();
}
```

This is presentation-only (no controller change). **Fallback if the coordinator prefers
zero AuthFlowSheet edits:** call `authSheet.openPicker()` from the Connect button ‚Äî the
picker preselects index 0 (matrix is the sole interactive family in practice). I recommend
the small extension for the more direct UX; it is confined to `AuthFlowSheet.qml`, tagged,
and additive. **No `AuthFlowController` change is needed either way.**

**bindProfile decision (open Q1):** recommend passing **empty** `bindProfile` for the
Channels connect ‚Äî the new Matrix account lands **unbound**, matching the shipped model
where rooms are pinned to sessions *explicitly* per-room (B4/B6, already shipped via the
RouteDialog on the ChannelsPage). Binding the whole account to one profile is a heavier
policy choice; if the coordinator wants it, the honest options are "bind to the active
default profile" or "offer a profile picker in the sheet". Defer unless directed.

### B2 ‚Äî Membership surfacing at v29 (GUI + TUI)

**Reality:** there is **no room-membership / ConvList push event** on the wire at v29
(the node events are roster/fleet/transport/approval/download/catalog/resync). An
auto-accepted, *unpinned* joined room creates **no** roster or fleet delta ‚Äî so the app
cannot be *pushed* about a newly joined room. Appearance is therefore refresh-driven.

**Honest surface (MVP):**
1. **Prompt appearance on connect.** On `TransportChanged` reaching `connected` for a
   transport (B5 path), `SubscriptionManager` also calls
   `transportRepository->refreshConversations(tx)` so freshly-joined rooms populate on
   connect rather than only on manual expand. Latency for rooms joined *after* connect =
   next refresh trigger (row expand / Channels-tab focus, both already wired). This is the
   honest latency; a dedicated conversations/membership event is a node follow-up (open Q3).
2. **"New" affordance.** When a `ConvList` refresh returns a conversation id not present in
   this transport's previously-known set, badge that room "new" until the user views it.
   The **membership itself comes from the node** (ConvList) ‚Äî the app only tracks
   *"which rooms the operator has already seen"*, which is presentation state, not domain
   re-derivation. Kept **in-memory per session** in `TransportRepository`
   (`QHash<QString, QSet<QString>> m_seenConversations`), exposed as
   `Q_INVOKABLE bool isNewConversation(tx, id)` via the `DaemonTransportRegistry`/
   `ITransportRegistry` seam; a room clears "new" once rendered/expanded. GUI: a
   `Kit.Chip { text: qsTr("new"); tone:"accent" }` on the room row. TUI: a `‚ú¶ new` token on
   the room line. This keeps GUI+TUI parity through the shared model.
3. **Auto-accept policy copy ‚Äî DEFERRED read, honest note.** The
   `[matrix].auto_accept_invites` value is **not queryable over the wire** at v29
   (`ConfigGet`/`NodeConfigView` is `Unsupported`/opaque per the v29 record). Asserting a
   specific policy value would violate render-honesty. So: (a) add a neutral,
   behavior-level note in the Channels "Accounts" section ‚Äî *"Room invites are handled by
   the node; newly-joined rooms appear here automatically."* (a statement about what the UI
   shows, not a claim about the config value); (b) record a **node-first follow-up**: a
   wire-queryable read of the auto-accept policy so the app can show the actual
   read-only value. Do **not** fake it. (open Q3)

### B5 ‚Äî Live presence via `TransportChanged` (GUI + TUI)

**Flow:** event ‚Üí dispatch ‚Üí repository (apply payload) ‚Üí presence projection ‚Üí UI, all
reactive, replacing the refresh/poll-derived placeholder.

1. **Codec facade** (`node_api_codec.h` + `decode_responses.cpp`, mine): add
   `DecodedNodeEvent::Kind::TransportChanged` + fields `QString transport`, `QString
   connection`, `QString presence`, `bool hasPresence`. Decode the
   `node_event_transport_changed_m_c` arm, mapping the generated `connection_state_r`/
   `presence_state_r` via `connectionStateName`/`presenceStateName` (expose these two
   helpers through `node_api_codec/internal.h`; today they are file-local in
   `fs_fleet_channels.cpp`).
2. **Repository** (`repositories.{h,cpp}`, mine): add
   `TransportRepository::applyTransportChanged(tx, connection, presence, hasPresence)`. It
   upsert-patches the single cached row's `connection`/`presence` (preserving
   family/displayName/boundProfile), then emits `instancesRefreshed()`. If the transport is
   **not yet cached** (e.g. brand-new account before the first `TransportInstances`), fall
   back to `refreshInstances()` (authoritative full fetch). Mirrors the
   `ModelRepository::applyDownloadProgress` precedent.
3. **Dispatch** (`subscription_manager.{h,cpp}`, mine): give `SubscriptionManager` a
   `TransportRepository*` (via a tagged `setTransportRepository()` setter ‚Äî the repo is
   constructed *after* the manager in the graph, so a setter beats reordering the ctor).
   In `applyEvent`, `case TransportChanged:` ‚Üí `m_transports->applyTransportChanged(...)`
   and, when `connection == "connected"`, `m_transports->refreshConversations(tx)` (B2 ¬ß1).
4. **Presence + UI:** `DaemonPresenceService` already rebuilds on `instancesRefreshed` and
   emits `presenceChanged(tx)` ‚Üí **no presence-service change**. GUI `ChannelsPage.qml`
   status dot + the connection text already re-read on `Presence.onPresenceChanged(tx)`;
   the TUI Channels page already re-renders on `presenceChanged`. So both surfaces light up
   with **no per-surface code change** ‚Äî the reactive seam pre-exists. The roster/sidebar
   account nodes that show a status dot (wave-1 merge) ride the same presence signal.

### F5 ‚Äî `FleetChanged` ‚Üí live fleet/roster refresh (GUI + TUI)

1. **Codec facade** (mine): add `DecodedNodeEvent::Kind::FleetChanged` (reuse the existing
   `rev` field); decode the `node_event_fleet_changed_m_c` arm in `decodeEventsPage`.
2. **Dispatch** (mine): `case FleetChanged:` ‚Üí `m_fleet->refreshTree("fleet")`. Direct
   (not debounced) ‚Äî the node emits `FleetChanged` at meaningful supervision transitions,
   not per token; it is the same `refreshTree` already called on every roster delta, so no
   new storm risk. (Optionally coalesce behind a short timer if the coordinator wants it ‚Äî
   open Q2.)
3. **UI:** `FleetRepository::treeRefreshed()` already drives `DaemonFleetTree` (GUI Fleet
   page + sidebar fleet nodes) and the dashboard counters ‚Äî **no per-surface change**. The
   fleet/roster surfaces update live without a manual refresh, which is exactly the gap.

## 3. Seam agreement with the app-delegation stream (F3/UnitNode)

The delegation stream owns the **`UnitNode` field decode** (the new `lifetime` and `engine`
optionals from F3) ‚Äî i.e. `codec_detail::decodeUnitNodeStruct(...)` (in
`node_api_codec/fs_fleet_channels.cpp`) and the `DecodedUnitNode` struct fields in
`node_api_codec.h`, plus the `Caps`/notice surfaces.

**This stream owns the `FleetChanged` event-to-refresh path**: the
`DecodedNodeEvent::Kind::FleetChanged` enumerator + fields in `node_api_codec.h`, its
decode arm in `decodeEventsPage` (`decode_responses.cpp`), and the `SubscriptionManager`
route to `FleetRepository::refreshTree`.

**Exact non-overlap:**
- Different structs in the shared header: I append to **`DecodedNodeEvent`**; they append to
  **`DecodedUnitNode`**. Kept as separate append-only blocks ‚Üí no textual conflict.
- Different functions in the shared codec: I edit **`decodeEventsPage`**
  (`decode_responses.cpp`); they edit **`decodeUnitNodeStruct`**
  (`fs_fleet_channels.cpp`). Different TUs.
- Contract: **on `FleetChanged` I only trigger `refreshTree` ‚Äî I never read/interpret
  `UnitNode.lifetime`/`engine`.** The subsequent `Tree` refetch decodes those fields
  through *their* mapper. So the enriched fields flow to the UI via their decode + my
  refresh trigger, with no shared mutable state. If both streams touch
  `node_api_codec/internal.h` (I expose `connectionStateName`/`presenceStateName`; they may
  add unit-field helpers), the additions are in distinct regions of the file ‚Äî append-only.

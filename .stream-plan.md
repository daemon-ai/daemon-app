## 0. TL;DR of what is actually true (grounded in code)

- The wire already carries, for a `ToolFinished` event **and** for a journal `ToolResult` block:
  `ok`, **`summary`**, and optional **`detail = { kind, body }`**.
  - `summary` is **the full tool output content** — `tool_result_view.summary = outcome.result.content`
    (daemon-node `crates/engine/daemon-core/src/engine/views.rs:104-111`). This is where the fs
    **diff text**, the shell/execute **stdout+stderr**, the fs **listing text**, and the
    web-extract **page content** actually live. It is **not** a short label.
  - `detail.body` is **JSON UTF-8** for every native tool (not CBOR, despite the protocol comment).
    ACP / stream-json adapter tools use CBOR, but those are foreign-agent paths; native tools use
    `serde_json::to_vec`.
- The client **decodes both today only on the live `AgentEvent` path**, and even there it keeps only
  `detailKind`/`detailBody` in `DecodedAgentEvent` (drops `summary`), then **flattens the event to
  `{callId,status}`** at `DaemonTurnEngine::toIngestEvent()` — so nothing reaches the renderer.
- Journal decode (`decodeTranscriptBlock`) **drops** `ToolCall_detail` / `ToolResult_detail`
  entirely (the wire structs carry them; the mapper never reads them).
- The SQLite cache table has **no** detail columns; the cold-start re-projection
  (`CachedSessionStore::content()`) emits a `tool` fence with only `{callId,name,argsSummary,status}`.
- **The renderers already exist and already consume flat view keys.** `ToolCallBlock.qml` +
  `transcript_render.cpp::emitToolBody` route by a **renderer discriminator** `detailKind ∈
  {ansi-stream, pty, diff, search-results, image, generated-image}` and read flat keys
  `stdout/stderr/diff/hits/imageUrl/body`. `SearchResultsList.qml` already renders **clickable**
  hits (`Qt.openUrlExternally`). `ImageBlock.qml`/`Lightbox.qml`, `DiffBlock.qml`, `AnsiText.qml`
  all exist. `agent_ingest.cpp::copyToolDetail` already lists exactly those flat keys.

**Therefore the work is a plumbing + one small projection function + persistence columns, not new
renderers** (one new fs-listing renderer is the only genuinely missing view, and it is optional —
see §6). The single missing piece of logic is a **node-kind → renderer-discriminator projection**
that turns `(kind, body-JSON, summary)` into the flat view keys the renderers already read.

## 1. Architecture decision

**Project late, in the shared view-model `buildToolView()`.** Carry the *raw* wire fields
(`detailKind` = node kind, `detailBody` = JSON bytes, `summary` = full content) unchanged through the
event stream, the ingest, the SQLite cache, and the journal replay. Do the node-kind→renderer
mapping in exactly one place: a new pure function `be::projectToolDetail()` invoked by
`be::buildToolView()` (BlockEditor `core/agent_block.cpp`).

Why this seam:
- `buildToolView()` is the single shared C++ transform behind **both** surfaces
  (GUI `block_model.cpp::buildToolData` → `ToolCallBlock.qml`; TUI
  `transcript_render.cpp::emitToolCall/emitToolBody`). Projecting there gives GUI+TUI parity for
  free and satisfies the AGENTS "one shared view-model" rule.
- Persistence stays layering-clean: `src/core/daemon/cached_session_store.cpp` only has to
  **serialize the raw fields** into the tool fence JSON; it does **not** need to depend on the
  BlockEditor renderer vocabulary (avoids a `src/core → src/DaemonApp` dependency).
- Raw storage is faithful: the mapping can improve later without a cache rebuild.

Rejected alternative ("project early": emit flat keys from `toIngestEvent` and store flat keys):
would force the mapping vocabulary into `src/core/daemon` (cold-start projection) and into the Turn
engine, duplicating the table across layers. Keep the mapping in one BlockEditor function.

Round-trip note: block metadata is serialized to a JSON fence (`serializeAgentBlock`) and re-parsed
(`applyAgentBlockFromFence` → `parseAgentBlockMetadata`, which merges *all* keys, nested arrays
included). A `QByteArray detailBody` becomes a base64 **string** after that round-trip, so
`projectToolDetail` must accept `detailBody` as either `QByteArray` (live) or base64 `QString`
(reloaded). `summary` is a plain string and round-trips as-is.

Projection rules encoded in `projectToolDetail`:
1. Default `body = summary` and no renderer `detailKind` (routes to the plain-mono text sub-renderer,
   which shows the full tool output — an immediate improvement over one-line summaries for **every**
   tool, even ones with no typed body).
2. Kind-specific overrides in the table above set the renderer `detailKind` and the specialized
   key (`diff`, `stdout`+`exitCode`, `hits`, `imageUrl`). `hits` maps `snippet`→`snippet`
   (matches `SearchResultsList.qml` and `emitSearchResults`, which read `title/url/snippet`).
3. Empty `summary` **and** empty typed payload → return `{}` (no detail body; header-only card,
   unchanged from today).

`todo` and `clarify` never reach `projectToolDetail`: `todo` is intercepted upstream (§5.1); the
`clarify` *variant* is driven by host-gate metadata, and the clarify tool detail is redundant (§6,
optional).

## 9. Risks, open questions, and scope-vs-reality contradictions

- **R1 — "shell/execute stdout" and "fs diff" are in `summary`, not `detail.body`.** The brief lists
  these as detail kinds; in reality the typed `detail` for `shell`/`execute_code` carries only
  *lengths*+`exit_code`, and `fs` carries only `{op,path,count}`. The actual bytes ride in
  `summary` (= `ToolResult.content`). **The plan uses `summary` as the content source** for these.
  This works, but note the client must now carry `summary` (a new field on the live event decode).
- **R2 — `summary` can be large.** Full stdout / large diffs land in the transcript markdown fence
  and the SQLite `summary` column. This grows the cached markdown. Acceptable (it is how message/
  reasoning content already persists), but consider a soft cap / lazy expand if outputs are huge.
  Open question: do we want a size guard before embedding `summary` in the fence?
- **R3 — browser screenshots are a node-local path, not bytes/URL (thin-client problem).**
  `browser_screenshot` detail is `{path}` — a filesystem path **on the node**. When the node is
  remote, the app cannot load it; `file://{path}` only works co-located (desktop) and never in WASM.
  **Open question / likely contract gap:** rendering screenshots properly needs the node to serve
  image bytes or a fetchable URL (that would be a daemon-node change, out of this frozen-wire scope).
  **Plan:** route to `image` with `file://{path}` (works co-located), and **degrade gracefully** —
  when the image fails to load, `ImageBlock` should show the path text (verify/ensure fallback).
  Flag to the program that full remote screenshot support is a node-side follow-up.
- **R4 — renderer `detailKind` vocabulary is client-invented, not the node's.** `diff`,
  `search-results`, `ansi-stream`, `image` are renderer discriminators; the node emits tool-name
  kinds. The mapping (§3) is the contract. `ToolStarted` also stashes a node-kind `detailKind` in
  metadata; ensure `buildToolView` always re-derives the renderer value so a stale node kind never
  reaches the `switch`.
- **R5 — `detailBody` base64 round-trip.** Live path passes `QByteArray`; reloaded fence passes a
  base64 `QString`. `projectToolDetail`/`buildToolView` must handle both, else reloaded tools render
  without detail. Covered by test #3.
- **R6 — foreign-agent CBOR kinds.** ACP / stream-json tools encode `detail.body` as **CBOR**, not
  JSON, and use kinds like `tool_result`/tool-title. `projectToolDetail`'s JSON parse will fail →
  falls to the text default with `summary`. Acceptable (still better than one-line). Do **not**
  attempt CBOR decode of detail bodies in this stream.
- **R7 — `todo` body is JSON (already handled).** Confirmed the existing `todoItemsFromDetail`
  JSON parse matches the node (`serde_json::to_vec`); no change, keep the interception.
- **Open question — fs `op:"list"` output format.** The `summary` listing text format is whatever
  `fs list` emits; a "mini-tree" needs to parse it. If the format is not reliably parseable, ship the
  text sub-renderer (full listing) and defer the tree. Decision recorded in §6 (text fallback OK).
- **Open question — surface `exitCode`/`count` in the GUI header?** The TUI plan adds an `exit N`
  line; decide whether the GUI `ToolCallBlock` header shows `exitCode`/`count` chips or leaves them
  in the body only. Non-blocking; default to body-only for parity simplicity.
